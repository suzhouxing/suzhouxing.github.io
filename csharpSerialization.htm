C#.NET Serialization: Examples and Comparison

1 Introduction

Serialization is the process that write data into a stream to preserve state of an object, and deserialization is to recover the state of an object by reading a stream of data.

The stream can be local files, memory and network flow.

The data can be in various formats such as binary, json, xml and plain text.

There are many ways to serialize/deserialize object into different streams with different formats in C#.NET. To my surprise, some third party libraries like protobuf-net is over 10x faster than the poor performance .NET build-in methods. As the benchmark page in protobuf-net is marked “Updated Feb 4, 2010”, I was wondering that, after 5 years, has the .NET improved its performance?

The code in next chapter can be a brief example to show the basic usage of:

The code patterns of methods above look really alike. They use Serializable, DataContract or ProtoContract attributes to mark the class or struct, and use DataMember, ProtoMember, XmlAttribute or XmlElement to mark fields or properties in the class.

There are some tips below:

l  Most methods need (empty) default constructors.

In most cases, you should leave the default constructor empty because they might be called by the deserializers. If you add 4 elements to a LinkedList in the default constructor, you may find out that there are 4 more elements than source stream in it after deserialization!

l  ProtoContract and ProtoMember can be replaced by DataContract and DataMember if there is no advanced usage.

This makes it very easy to migrate your serialization from DataContractSerializer to protobuf-net. But one thing to notice is that, you must set the Order in DataMember which maps to the id in ProtoMember.

l  Json.NET seems not to flush the stream after serialization.

In the early version of my benchmark code, tests on Json.NET always failed. I checked out the output stream and find it is in a mess. So if you want to test serialization and deserialization together, you may need to flush the stream by hand. In other cases, it will not bother anything.

l  BinaryFormatter will record metadata of the object.

Sometimes you may find a strange thing that serializing to a binary file result in a greater size than to a json or xml file! For a programmer with C/C++ background this is really confused. The fact is that BinaryFormatter will write many metadata with the object. The C style fwrite and fread equivalent might be the BinaryWriter and BinaryReader. In the latter case, you need to manipulate every data field by hand.

l  Only BinaryFormatter can deal with the cyclic reference.

In extreme conditions like the two class: class A { B b; } and class B { A a; }, if a.b.a = a, most serializers are not able to recognize the cyclic reference.

        public static readonly string[] s = new string[]{

            "1","12","123","1234","12345","123456",

            "1234567","12345678","123456789","1234567890"

        };

 

        [DataMember(Order = 1)]

        public int i;

        [DataMember(Order = 2)]

        public Dictionary<int, string> d;

        [DataMember(Order = 3)]

        public double[] a;

        [DataMember(Order = 4)]

        public LinkedList<char> l;

        [DataMember(Order = 5)]

        public HashSet<long> h;

        [DataMember(Order = 6)]

        public List<int> t;

        [DataMember(Order = 7)]

        public Queue<short> q;

        [DataMember(Order = 8)]

        public BitArray b;

 

        public A() { }

 

        public A(int i) {

            this.i = i;

 

            d = new Dictionary<int, string>();

            for (int k = 0; k < i; ++k) {

                d[k * 77777777] = s[k % s.Length];

            }

 

            a = new double[i];

            for (int k = 0; k < i; ++k) {

                a[k] = k * 7777.7777;

            }

 

            l = new LinkedList<char>();

            for (int k = 0; k < i; ++k) {

                l.AddLast('s');

            }

 

            h = new HashSet<long>();

            for (int k = 0; k < i; ++k) {

                h.Add(k * 77777777);

            }

 

            t = new List<int>();

            for (int k = 0; k < i; ++k) {

                t.Add(k * 77777777);

            }

 

            q = new Queue<short>();

            for (int k = 0; k < i; ++k) {

                q.Enqueue((short)(k * 77777777));

            }

 

            b = new BitArray(i);

            for (int k = 0; k < i; ++k) {

                b[k] = (k % 2 == 0);

            }

        }

    }

2.4 Analysis

2.4.1 Compatibility

The BinaryFormatter and DataContract are the most compatible approaches to serialize/deserialize objects. On the other hand, XmlSerializer seems to work on least types.

2.4.2 Performance

For most objects with a simple layout of an array or a container in system library, the BinaryFormatter is the most efficient way to achieve a serialization.

But in real world, with an object of a complex class, the protobuf-net may be the best choice.
